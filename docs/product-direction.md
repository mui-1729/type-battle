# プロダクト方針

## 目標

まずは友人・知人など内輪で遊べるオンラインタイピング対戦ゲームを作る。将来的には Web に公開し、知らない人同士でも遊べるサービスへ拡張できるようにする。

## 段階

### Stage 1: Local MVP

ローカル環境で 2 人対戦が成立する状態。

- room code で参加できる。
- 同じ文章で対戦できる。
- 進捗、WPM、正確率、結果が同期される。
- 2 ブラウザ E2E で基本フローを確認できる。

### Stage 2: Private Beta

内輪で URL を共有して遊べる状態。

- ゲストのニックネーム参加で遊べる。
- room code を知っている人だけが参加できる。
- 最低限のデプロイ環境がある。
- 軽い rate limit とログがある。
- 不具合を GitHub Issues で集められる。

この段階では、ランキング、ログイン、ランダムマッチは必須ではない。

### Stage 3: Public Beta

知らない人にも試してもらえる状態。

- 利用規約、プライバシー、問い合わせ導線を用意する。
- 荒らし対策、通報、ブロック、名前の制限を追加する。
- サーバー負荷、エラー、同時接続数を監視する。
- ランダムマッチや公開ロビーを慎重に追加する。
- 不正タイピングや bot 的な挙動を検知する。

### Stage 4: Public Service

継続的に遊ばれる Web サービスとして運用する状態。

- ログイン、戦績、ランキング、レートを追加する。
- 複数人対戦、観戦、フレンド対戦などを拡張する。
- 継続的なコンテンツ追加と運用体制を作る。
- 本番障害対応、バックアップ、コスト管理を整える。

## 最初から意識すること

内輪向けに作る段階でも、将来の公開を妨げないように次は最初から入れる。

- guest id を発行し、プレイヤーを nickname だけで識別しない。
- room code は推測しにくくする。
- nickname は長さ、禁止文字、HTML escape を制御する。
- 重要な勝敗判定はサーバー側で行う。
- event payload は型定義で共有する。
- エラーや切断をログとして残せる形にする。

一方で、次は MVP では作り込みすぎない。

- 完全な認証基盤
- 永続ランキング
- 高度な通報・モデレーション
- 大規模マッチメイキング
- 課金

## 成功条件

### Private Beta の成功条件

- 友人同士で room code を共有し、5-10 試合連続で大きな同期崩れなく遊べる。
- 操作説明なしでも、room 作成、参加、開始、再戦が理解できる。
- 切断やリロードでゲーム全体が壊れない。
- 不具合報告から再現条件を追えるログがある。

### Public Beta へ進む条件

- Private Beta で主要な対戦フローが安定している。
- デプロイ、監視、ログ確認の手順がある。
- 荒らし、名前、過剰アクセスへの最低限の対策がある。
- 公開時の問い合わせ先と注意書きが用意されている。
